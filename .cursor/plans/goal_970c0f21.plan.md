## Goal

Replace the manual prompt in [README.md](README.md) (lines 3-25) with a discoverable, repeatable skill that bakes in two real constraints from the live form:

1. **Stale-file trap** — freelancermap keeps every prior `Resume.pdf` as a tickable attachment. Skill uploads fresh under a stable name and verifies exactly one (the new) attachment is checked.
2. **Stop before submit** — every send is human-eyeballed; never exceed the on-page quota.

## Storage in this repo (version-controlled)

Per application, the skill overwrites:

- `resume_adapted.md` — adapted markdown
- `resume_adapted.pdf` — built by `make`
- `Resume-Miguel-Dias.pdf` — the upload-stable copy
- `applications.csv` — appended one row

Then commit. `git log -- resume_adapted.md` is the per-job archive. No per-application directories, no `url.txt`, no committed screenshots.

`applications.csv` columns: `date,title,company,contact,url,language,pdf_sha256,status,message`.

## Skill shape

`~/.cursor/skills/apply-to-freelancermap-job/SKILL.md` — single file. Inline: workflow checklist, EN+DE message template, adaptation rules, upload+verify logic, the `python3 -c '...'` CSV-append snippet.

## Workflow the skill executes (per URL)

1. WebFetch URL → `{title, company, contact, description, language}`. Detect DE if `lang="de"` or >5 German stopwords.
2. Adapt `Resume.md` → `resume_adapted.md` per inline rules. Adapt inline message template (EN/DE).
3. `make` → `resume_adapted.pdf`. `cp` to `Resume-Miguel-Dias.pdf`. Capture sha256 + mtime.
4. Open URL in browser. Halt if not logged in. Read `Applications remaining: N/10`; halt if `0`.
5. Type message into `Message *` textarea. Upload `Resume-Miguel-Dias.pdf` via `<input type="file">`. On upload failure: halt with the absolute path so I can drag-drop manually.
6. Verify exactly one attachment checked, name matches, mtime matches. Untick stale `Resume*.pdf`. Halt on mismatch.
7. Append row to `applications.csv` (`status = "Prepared, awaiting submit"`). Report summary. STOP before `Send application`.

## How beads is used here

- One bead `in_progress` at a time. No fan-out.
- Every bead created with `-d "...Done when: <observable criterion>"`. No DoD = don't create the bead.
- Before `bd close`: append a result note (`bd update <id> --notes "..."`).
- After `bd close`: git commit referencing the bead id (`CV-xxx: <title>`).
- Mid-task discoveries → new bead with `--deps discovered-from:<current-id>`, don't sidetrack.

For this scope: one ephemeral Cursor agent per bead, no queues, no personas. Parallel agents would contradict the "one URL at a time, human-eyeballed" constraint.
